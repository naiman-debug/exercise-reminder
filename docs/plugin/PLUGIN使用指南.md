## 🤔 怎么选：Plugin 还是 MCP？

### 简单判断法

```
┌─────────────────────────────────────────────────────────┐
│  你想做什么？                                            │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  📝 审查代码          →  说："帮我审查这个PR"           │
│                         (自动用 code-review)            │
│                                                         │
│  🎨 设计前端界面      →  说："设计一个登录页面"         │
│                         (自动用 frontend-design)        │
│                                                         │
│  ⚡ 开发新功能        →  说："帮我实现用户登录功能"     │
│                         (自动用 feature-dev)            │
│                                                         │
│  🔧 GitHub操作         →  说："创建一个PR"              │
│                         (自动用 GitHub MCP)             │
│                                                         │
│  📚 查询文档          →  说："React的useEffect怎么用"   │
│                         (自动用 context7)               │
│                                                         │
│  🚫 设置规则          →  说："用hookify创建规则"        │
│                         (需要用 /hookify:hookify)       │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 核心区别

| | Plugin (命令) | MCP (工具) |
|---|---|---|
| **触发方式** | 输入 `/命令名` | **自动**，我帮你调用 |
| **使用难度** | 需要记住命令 | **零学习**，自然对话 |
| **适用场景** | 特定复杂流程 | 日常操作 |
| **示例** | `/code-review` | "帮我审查代码" |

## 🎯 使用场景

### 1️⃣ code-review - 代码审查

**不用记命令，直接说：**

| 你说 | 我会做 |
|------|--------|
| "帮我审查这个PR" | 自动调用 code-review |
| "看看 #123 PR 有什么问题" | 分析代码并提供反馈 |
| "这个代码质量怎么样？" | 给出改进建议 |

**如果你想手动控制：**
```
/code-review
```
然后按提示操作

---

### 2️⃣ frontend-design - 前端设计

**直接说需求：**

| 你说 | 我会做 |
|------|--------|
| "设计一个登录页面" | 生成高质量前端代码 |
| "做一个商品卡片组件" | 创建生产级UI组件 |
| "我要一个响应式导航栏" | 输出完整HTML/CSS/JS |

---

### 3️⃣ feature-dev - 功能开发

**直接描述需求：**

| 你说 | 我会做 |
|------|--------|
| "帮我实现用户认证功能" | 分析代码库并实现功能 |
| "添加一个支付模块" | 理解架构后开发 |
| "优化这个API的性能" | 找出问题并修复 |

---

### 4️⃣ agent-sdk-dev - 创建Agent应用

**直接说：**

| 你说 | 我会做 |
|------|--------|
| "创建一个新的Agent应用" | 引导你完成搭建 |
| "初始化一个SDK项目" | 自动配置和安装 |

---

### 5️⃣ hookify - 设置规则

**这个需要用命令：**

```
/hookify:hookify    # 创建规则
/hookify:list       # 查看所有规则
/hookify:help       # 获取帮助
/hookify:configure  # 配置规则
```

**常用场景：**
- 防止我说某些话
- 设置对话规则
- 限制某些话题

---

### 6️⃣ GitHub 操作 (MCP)

**直接说就行：**

| 你说 | 我会做 |
|------|--------|
| "查看我的仓库列表" | 调用 GitHub MCP |
| "创建一个新分支" | 自动执行 git 操作 |
| "提交这次修改" | 帮你写 commit 并 push |
| "创建一个PR" | 自动创建 Pull Request |

## 🎓 进阶技巧

    1. 用 frontend-design 设计界面

## ❓ 常见问题

**Q: 我需要记住所有命令吗？**
A: **不需要！** 大部分时候直接说需求就行，我会自动选择合适的工具。

**Q: 什么时候必须用命令？**
A: 只有 hookify 相关功能需要用 `/` 命令，其他都可以直接对话。

**Q: MCP 和 Plugin 会冲突吗？**
A: 不会。它们各有分工，我会根据场景选择最合适的。

**Q: 我怎么知道你用了哪个工具？**
A: 我会在回复中说明，比如 "(调用 code-review)"。
